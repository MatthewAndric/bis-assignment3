Angela Mao [s3857679] [s3779854@student.rmit.edu.au]

I have found that learning Unity’s interface and basic functions was not as hard as I thought as there are plenty of online resources that offer help into how to use it. It was unexpectedly not complicated and easy to learn as Unity simplified many of the tasks for me and was straightforward.

The only thing I found difficult and being a huge obstacle would’ve been the Unity Teams feature of Unity. This feature allowed the group to collaborate on the same project through using a push and commit system, and it would sync with each instance of the project on our systems. However sometimes it found itself to be unreliable and buggy which added a level of difficulty for myself.

The challenges I were expecting were going to be the eventual time and workload constraints as the semester went on, with other subjects and personal life to juggle. To address this, I tried as hard as possible to stick to the schedule of contributing some work every week. I had overcome this challenge by sticking to this schedule and successfully implementing features that helped keep me on top of things.

Another challenge would be having trouble programming and knowing how to use unity, as I only have basic programming skills and C# was a whole new programming language to learn. I addressed this challenge with my previous programming knowledge and consulting tutorials on youtube and looking for help on google whenever I had trouble. I did this knowing Unity has a supportive community and many resources to accomplish tasks in Unity. With my research, I overcame the programming obstacle for this particular project.

These two challenges stretched me as it meant I had to be more concise with my time management and research for long hours, sometimes with no beneficial outcome. However, hours spent on this project gave me experience in game development which made it well worth the time.

Halfway into the development of the game, plans changed as someone in our group had withdrawn from the class, leaving us at a disadvantage. This meant that work would have to be increased among each member to accommodate for this change. Instead of focusing on one or two MVFs, I had to end up focusing on a third one as it tied heavily into one of my other MVFs I was working on. This resulted in changing the time allocated to each MVF and ultimately affected the quality of what was done. Some MVFs were cut short, for example, having a wider variety of enemy attacks.

In hindsight, I probably would have just made less MVFs and workload and just tried to fit five MVFs that we would have spent most of our time on, so that the quality was better. I also would have made sprites as we finished per level, so we didn’t end up with extra unused sprites.

I found the timetable to be unrealistic as in reality, I had put hours into working on MVFs that weren’t my own, which did not happen. I had only worked on my MVFs alone and left the other MVFs to the members that were in charge of them. However, the time put into the MVFs that I did were accurate.

All of the tools and technologies that I used worked out as expected, except for Unity Teams as I mentioned earlier. Some project risks we had predicted that were encountered while working on the project were time restrictions and the final product not reflecting our intended vision (due to time constraints.)

Ziqiang Lin [s3792327] [s3792327@rmit.student.edu.au]

For this project to come up with ideas is not so hard. But implementing them is kind of difficult. There is always a collision between object and bug between code that is made by different people. The unexpected things are that if people are using different visions of Unity to work together, the code would not work properly when combining which can cause a lot of work to fix and a lot of code changing.

Since the big part of the project is pure coding. I was expecting to spend a lot of time learning how to code with Unity. But I find that there are a lot of resources online for learning and also a documentation website that explains the function of the method and code.

From the plan and timetable our group made before, I find it not so realistic to follow since the time to do each feature can vary depending on the element it relates to. And also throughout the semester, there are times when multiple assignments are due at the same time, this made the group focus less on the project. I think a better way to make a timetable would be to make adjustments every few meetings so everyone would have a rough idea of where the project is heading and make them not lost in what they need to work on.

For this project, we use the Unity Team, which is not a very well designed function by Unity. Every time a team member makes a change in the project, for others to see that change usually requires multiple refreshes. And if others are not aware of the change, the stuff they commit would overlap and the application would only save the newest one. To address the issue, we keep communication when making any changes, which allow all the group members to understand what others are doing and what each code is for.

From this project, I learn how to code with C# and more importantly how to work with a team remotely. Through the use of GitHub and Microsoft Teams, I learned how to share work with each other and how to communicate effectively when something needs to be done.
